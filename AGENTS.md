# Agent Guidelines

> **Expo HAS CHANGED** — Read the exact versioned docs at https://docs.expo.dev/versions/v56.0.0/ before writing any code.

## Commands

```bash
# Start the app
npm start               # Expo dev server (choose iOS / Android / web)
npm run ios             # Start with iOS simulator
npm run android         # Start with Android emulator
npm run web             # Start web version (localhost)

# Types
npm run typegen         # Regenerate API + Convex types (runs scripts/typegen.sh)
```

There is **no test runner, linter, or formatter configured in the frontend root**. The backend (`backend/convex/`) has its own AGENTS.md with NestJS/Convex commands and tests.

## TypeScript

- `tsconfig.json` extends `expo/tsconfig.base` with `strict: true`.
- Use explicit `type` imports when importing only types: `import type { Foo } from './foo'`.
- Prefer `type` over `interface` for object shapes (used consistently in this codebase).
- Use `as const` for readonly constant objects (e.g. `STORAGE_KEYS`).
- Avoid `any`; use proper generics or `unknown` if needed.

## Imports

Order and group imports as follows:
1. `React`
2. `react-native` components/hooks
3. Third-party libraries (navigation, icons, animation, etc.)
4. Internal modules (relative paths: `../theme/tokens`, `../store/authStore`, etc.)

Use single quotes. No `.js` extensions on relative imports.

Example:
```tsx
import React, { useState } from 'react';
import { View, Text, StyleSheet } from 'react-native';
import { useNavigation } from '@react-navigation/native';
import { ChevronLeft } from 'lucide-react-native';
import { colors, spacing, fontSize, font } from '../theme/tokens';
import { AppButton } from '../components/ui/AppButton';
```

## Naming Conventions

- **Components / Screens**: `PascalCase.tsx`, exported as named functions: `export function HomeScreen()`.
- **Utility files**: `camelCase.ts`, exported as named functions/constants.
- **Types**: `PascalCase` for type names; `Props` for component prop types.
- **Navigation params**: `PascalCaseParamList` (e.g. `RootStackParamList`).
- **Constants**: `SCREAMING_SNAKE_CASE` for storage keys and magic strings.

## Formatting & Style

- Single quotes for strings and JSX attributes.
- Use `StyleSheet.create({})` at the bottom of the file.
- Inline styles (`style={{ color: foo }}`) are allowed **only** for dynamic values; static styles belong in `StyleSheet`.
- Use theme tokens for all design values: `colors`, `spacing`, `fontSize`, `font`, `radius`, `shadow` from `src/theme/tokens`.
- No trailing commas in objects/arrays (follow existing style).
- Section comments use `// ── Section name ──` dividers.

## Component Patterns

- Use function components (not class components).
- Memoize list items with `React.memo` when appropriate (see `OptionRow`).
- Use `Pressable` for tappable elements; apply `pressed` state styles via the callback.
- Add `* as Haptics` from `expo-haptics` and trigger light impact on button presses (guard with `Platform.OS !== 'web'`).
- Use `SafeAreaView` as the root of every screen.

## State Management

- Auth state lives in `src/store/authStore.tsx` using React Context + `useState`.
- Paywall/unlock state lives in `src/store/paywallStore.tsx`.
- Progress is no longer stored locally; it is persisted on the backend.
- Local state uses `useState`; effects use `useEffect`.

## Navigation

- Typed param lists are defined in `src/navigation/types.ts`.
- Screens receive `navigation` and `route` props typed against these param lists.
- Deep linking config lives in `App.tsx` inside the `linking` object.

## Data & Assets

- Static JSON data is loaded via `src/data/loaders.ts` (questions, vocab, clue words, guide, model tests).
- Content JSON files live in `src/data/json/` and are generated by Python scripts in `content/`.
- If content schema changes, update the corresponding Python builder and re-run `npm run typegen`.

## Error Handling

- Use `Alert.alert` from `react-native` for user-facing errors.
- Guard native-only APIs with `Platform.OS !== 'web'` checks.
- Avoid throwing uncaught errors in UI code; handle gracefully with fallbacks.

## Backend

The `backend/convex/` directory contains a separate NestJS + Convex backend with its own AGENTS.md, tests, linting, and CI. Do not mix frontend and backend tooling.

## Platform Notes

- The app targets **iOS, Android, and web** via Expo.
- Web-specific styles (e.g. `boxShadow`) should be cast `as any` and guarded with `Platform.OS === 'web'`.
- Use `react-native-web` compatible APIs only.

## Workflow & Contribution Conventions

Follow these on **every** task — they are how we ship.

### Plan first, then execute
- Non-trivial work starts as a spec in `docs/plans/*.md` with **locked decisions** (numbered, e.g. D-A…). Execute against the plan; if a decision must change mid-flight, update the plan doc.

### Batch into an integration branch (don't merge straight to `master`)
- Feature branches → PR into the current **integration branch** (e.g. `feat/next-changes`), not `master`.
- `master` only receives the whole batch once it's coherent and approved, via a single `integration → master` PR.
- Keep `master` releasable at all times.
- **Guard against drift:** if `master` gets hotfixes while an integration branch is open, merge `master` *into* the integration branch periodically so the eventual promotion stays a clean diff.
- Prefer **squash-merge** for the final `integration → master` PR (one clean commit); link the sub-PRs in its body so review can be incremental.

### Always keep a minimal CI gate on PRs
- Every PR must pass, at minimum: `npx tsc --noEmit` (clean) and the **i18n parity check** (en/fi key sets must match per namespace). Add/maintain a lightweight GitHub Action for this; never rely on manual checking alone.
- Run `npx tsc --noEmit` locally before opening or updating any PR.

### i18n rules (see also Data & Assets)
- UI **chrome** only goes through i18n (`src/i18n`, split per-namespace under `locales/{en,fi}/`). **Never** translate content (questions, answers, vocab, clue words, lesson names, descriptions).
- Keep `en.json` and `fi.json` namespaces at **full key parity** — a missing key must fail CI, not silently fall back to English.
- For bilingual data fields (`*_fi`/`*_en` titles), use `localizedPair()` from `src/i18n/content.ts` (app-language primary, other as subtitle).

### PR descriptions
- The `integration → master` PR is the consolidated record: write it as a **decision log + checklist**, listing open questions, deferred items (e.g. backend BE-x, content gaps), and links to sub-PRs — not just a changelog.

### Verify before promoting
- Before suggesting the `integration → master` PR: `tsc` clean, run the app (Expo web is fine) and walk the changed screens (incl. Suomi for any i18n work).
